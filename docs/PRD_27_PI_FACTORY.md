## 1. 목적·배경

마스터는 cafe.pi 외 **.pi 도메인 18개**(총 19개 — §2.5 확정 목록, 2026-07-19 마스터 확정)를 보유. 도메인마다 독립 프로젝트를 만드는 방식(리포 복제)은 유지보수 비용이 도메인 수에 비례해 폭증하므로, **cafe.pi를 모선(母船)으로 하는 멀티테넌트 팩토리**로 전환한다.

**전략 모델: 입주 → 검증 → 졸업 (인큐베이터)**
- **입주**: 신규 .pi 도메인 = DB 행 1개 + 도메인 연결 (목표: **1일 1사이트**)
- **검증**: 모듈 프리셋으로 MVP 운영, 시장 반응 실험 (실험 비용 1/20)
- **졸업**: 임계(MAU·매출) 도달 사이트만 독립 리포+전용 DB 분리
- **확장**: 타인 .pi 도메인 입주 SaaS화 (입주비·수수료 — 기존 요금 인프라 재사용) → "Pi 생태계의 Shopify" 포지션

### 1.1 접근 방식 비교 (결정 근거)

| 방식 | 신규 1건 | 유지보수 | 판정 |
|---|---|---|---|
| 리포 복제 ×20 | 며칠 | 패치 ×20회, 코드 분화 | ❌ |
| 모노레포 core 추출 | 반나절 | 양호 (전면 리팩토링 선행 수주) | 보류 (졸업 단계에서 재검토) |
| **멀티테넌트 팩토리** | **몇 시간** | **1곳 수정→20개 반영** | ✅ **채택 (2026-07-09 승인)** |

핵심 판단: 20개 도메인의 목적은 20개 풀 서비스가 아니라 **도메인 선점 가치 실현 + 빠른 실험**. 역방향 효과가 본질 — 한 사이트에서 검증된 기능·수익모델이 즉시 나머지 19개의 무기가 된다.


### 2.1 데이터 모델 — **확정 DDL 정본: `sql/173_site_factory.sql`** (da-ddl-guard 통과, 2026-07-09)

| 테이블 | 역할 | 핵심 |
|---|---|---|
| `sys_site_mst` | 사이트(테넌트) 마스터 | `site_cd` PK · `site_domain_nm` UNIQUE(Host 매칭 키) · `preset_cd`(COMMERCE/COMMUNITY/CONTENT) · `module_cfg_json` 오버라이드 · `use_yn` kill switch(기본 N — 실기기 검증 후 Y) |
| `sys_site_usr_map` | 사이트×사용자 Pi uid 매핑 | PK `(site_cd, pi_uid)` · `usr_id`→sys_user 전역 1행 · 활성 매핑 부분 인덱스 |

- **명명 결정**: 사이트 설정은 `sys_` 주제영역(선례 `sys_quick_menu`) — 신규 접두사 신설 대신 표준 준수(da-ddl-guard R2 통과). 시드: CAFE(활성)·SITEMAP(파일럿, 대기 — 2026-07-19 변경).

- **정체성 원칙**: 사람의 전역 불변 키 = `pi_username` (기존 철칙 그대로). `sys_user`는 전역 1행, 사이트별 scoped uid는 `usr_site_map`으로 흡수 → **어느 .pi에 가입해도 같은 계정·같은 Bean** = 네트워크 효과.
- **콘텐츠 테이블 site_cd 축**: `msg_room`·`mps_shop/item`·`brd_post` 등에 `site_cd TEXT NOT NULL DEFAULT 'CAFE'` 추가(기존 데이터 무손실). 조회 헬퍼에 사이트 필터 일괄 적용.

### 2.3 선결 기술 과제 — `NEXT_PUBLIC_*` 런타임화 ⚠️

`NEXT_PUBLIC_*`는 빌드 인라인이라 도메인별 분기 불가. 도메인별로 달라야 하는 클라이언트 값(OAuth CLIENT_ID·APP_URL·딥링크 도메인 등)은 **서버가 Host 기반으로 내려주는 사이트 컨텍스트**(RSC props 또는 `/api/site-config`)로 이전한다. `detectSandbox()` 런타임 판정 패턴 준용. **P1 최우선 작업.**

## 2.5 도메인 포트폴리오 및 온보딩 웨이브 (2026-07-19 마스터 확정, **총 19개**)

### 계층 구조 — 모선 → 파일럿(관문) → 분기 (2026-07-19 재편)

```
cafe.pi (모선) ─── 코드베이스·DB·인프라 소유 (Vercel 1프로젝트)
   │
   └── sitemap.pi (⭐파일럿 1호 + 네트워크 관문 — 2026-07-19 yea.pi에서 변경)
         │  · 멀티테넌트 인프라 자체를 검증하는 최소 사이트 (Host 분기·site_cd·usr_site_map·페르소나)
         │  · 이후 모든 사이트는 sitemap.pi에 등재된 뒤 개별 도메인으로 분기
         │  · 사이트 등재·온보딩 관리 UI(/admin/sites)는 sitemap.pi 소관
         │
         ├── W2: yea.pi(첫 응용) · gifticon.pi · barista.pi · bluemountain.pi
         ├── W3: seminar.pi · teamkorea.pi · expedition.pi · omok.pi · lan.pi
         ├── W4: dbms.pi · schema.pi · webserver.pi · was.pi · fondation.pi · yoda.pi
         └── W5: anakin.pi · youngrok.pi
```

- **파일럿 변경 사유** (2026-07-19 마스터 지시): sitemap.pi는 기능이 가장 얇아(디렉토리·대시보드·페르소나 — 결제·투표 없음) ① 멀티테넌트 인프라 순수 검증에 최적 ② 메인넷 심사 리스크 노출 면적 최소(게이트 0 관점) ③ 관문을 먼저 세우면 이후 모든 사이트가 처음부터 등재→분기 구조를 가짐.
- **"분기"의 의미**: 기술적 라우팅은 여전히 middleware의 Host 판정(§2 ①) — sitemap.pi는 **사용자 진입 관문**(네트워크 디렉토리)이자 **운영 관문**(사이트 등재·온보딩 관리)이며, 트래픽이 sitemap.pi를 경유하는 구조가 아니다. FR-P2(교차 노출 금지)는 그대로 — 개별 사이트에서 sitemap.pi 링크는 법적 고지 문서 내로 한정.

### ⭐ 수익 구조 대원칙 — Web3 크리에이터 이코노미 (2026-07-19 마스터 확정, 전 사이트 예외 없음)

**"누구든 정보를 제공하고, 그 정보로 유료 서비스가 발생하면 수익은 정보제공자의 것. 플랫폼은 일부 수수료만."**

1. **크리에이터 퍼스트 분배**: 유료 결제(수강료·입장료·콘텐츠 열람·상품 판매·응원·팁)의 대부분은 정보제공자(강사·저자·판매자·개최자·크리에이터) 수익. 플랫폼은 수수료만 수취(요율은 `bean_fee_plan`/fee 정책 단일 출처 — 하드코딩 금지).
2. **정산 = 검증된 A2U 자동정산 재사용**: MPS `settleOrder`(release_txid 멱등·cron) 패턴을 전 사이트 표준으로. 보상성 A2U의 관리자 승인 게이트는 유지.
3. **오픈 참여**: 교육·콘텐츠 사이트는 플랫폼 자체 콘텐츠가 아니라 **누구나 강사·저자로 등록해 판매하는 오픈 마켓플레이스** 구조.
4. **구독 번들 수익 배분**: 통합 구독 수익은 소비(열람·수강) 비중 기반으로 크리에이터에게 배분(스트리밍 로열티 모델 — 배분 로직은 P2 과제).
5. **플랫폼 고유 수익 예외**: 크리에이터 콘텐츠가 아닌 지면·인프라 수익(sitemap 광고·개설료·노출·부스팅·연장)만 플랫폼 직접 수익.
6. **어휘 절제**: 구조는 Web3, 표면 어휘는 심사 절제 — "토큰·탈중앙·수익 셰어링" 류 마케팅 문구는 게이트 0 감사 대상(Bean 순화 교훈 준용). 사용자 표면은 "내 수익·정산" 같은 실용 어휘로.

> 수익모델(R1~R3)은 cafe.pi 검증 프리미티브(개설료·입장료·노출·연장·구독·P2P 수수료·O2O 정산·건당 과금·팁·보증금)만 조합 — 신규 결제 인프라 0건. ⚠️ = 법무·규제 게이트 통과 전 수익화 금지.

## 2.6 프라이버시 설계 원칙 — "신원과 돈은 하나, 얼굴과 기록은 사이트별" (2026-07-09)

> 사용자 불안("내 정보가 여러 사이트에 공유되나?")에 대한 구조적 대책. 공유·분리의 경계선이 이 설계의 핵심.

### ⭐ 확정 요구사항 (2026-07-09 마스터 지시 — 필수, 예외 없음)

| FR | 요구사항 | 구현 귀결 |
|---|---|---|
| **FR-P1** | **같은 계정이지만 사이트마다 다른 별명·이름으로 활동할 수 있어야 한다** | `sys_site_usr_map.site_nick_nm`(+아바타) — 화면 표시명 조회는 항상 **사이트 페르소나 우선**, 미설정 시 전역 닉네임 자동 노출 금지(첫 진입 시 설정 유도). 마스킹 규칙([[username-masking-rule]])은 페르소나에도 동일 적용 |
| **FR-P2** | **사용자가 물리적으로 다른 사이트라고 인지할 수 있어야 한다** | 사이트별 고유 도메인·로고·컬러 테마·타이틀·OG 메타 완전 분리. **교차 브랜드 UI 노출 금지**(타 사이트 링크·배너·"cafe.pi에서 온 사용자" 류 표기 금지). 운영 주체 동일성 표시는 법적 고지 위치(약관·방침 문서 내)로 한정 — 일반 UI에는 노출하지 않음 |

| 구분 | 항목 | 근거 |
|---|---|---|
| **전역 공유 (불변)** | 계정(pi_username)·Bean 지갑·제재/1인1계정 | 지갑 분리는 오히려 불편·제재 분리는 어뷰저 은신처화 |
| **사이트별 분리 (기본)** | 표시명·아바타(페르소나)·활동 기록(게시글·후기·채팅) | 맥락 붕괴 방지·부캐 문화 수용 (site_cd 분리로 구조 보장) |

**구현 6종**: ① `sys_site_usr_map`에 `site_nick_nm` 등 페르소나 컬럼(sql/173 반영 완료) — 새 사이트 첫 진입 시 전역 닉네임 자동 노출 금지, 사용자가 "프로필 가져오기" 선택 시에만 공유 ② 활동 비공유 원칙 방침·화면 명문화 ③ **첫 로그인 1회 투명 고지**("계정·Bean 공유, 활동·프로필은 이 사이트에만") ④ **통합 관리 허브 = sitemap.pi** (아래) ⑤ 제재 공유는 선량한 사용자 보호 명분으로 약관 고지 ⑥ 개인정보처리방침 v1.2 예약 — "동일 운영자 서비스군 내 통합 계정" 고지(제3자 제공 아닌 동일 처리자 내 이용).

### ⭐ 통합 관리 허브 = sitemap.pi (2026-07-09 마스터 확정)

**"통합의 얼굴은 sitemap.pi 한 곳"** — FR-P2(개별 사이트의 브랜드 독립)와 통합 관리 필요성의 긴장을 도메인 분업으로 해소한다.

| 역할 | 내용 |
|---|---|
| **네트워크 디렉토리** | 전 사이트 목록·소개·바로가기 (.pi 생태계의 관문) |
| **내 사이트 대시보드** | 가입 사이트 목록·가입일·공유 항목 안내·**사이트 단위 연결 해제**(논리삭제) |
| **페르소나 관리** | 사이트별 별명·아바타를 한 화면에서 일괄 관리 (FR-P1의 관리 UI) |
| **통합 자산 뷰** | Bean 잔액·거래 내역(사이트별 필터)·통합 계정 설정 |
| **공식 고지 위치** | "동일 운영자 서비스군" 안내의 정본 — 각 사이트 약관·방침의 통합 계정 조항이 sitemap.pi를 링크 |

- 개별 사이트 UI에는 sitemap.pi 링크조차 최소화(법적 고지 문서 내 링크만) — FR-P2 준수.
- **웨이브 조정 이력**: W4 → W2 승격(2026-07-09) → **W1 파일럿 1호 확정(2026-07-19 마스터 지시)** — 관문을 먼저 세우고 나머지 전 사이트가 여기서 분기(§2.5 계층 구조).

## 3. 신규 사이트 SOP (목표: 1일 1사이트)

1. Pi Developer Portal 앱 등록 + 도메인 검증 (수작업)
2. Vercel 프로젝트에 도메인 추가
3. `/admin/sites`에서 사이트 행 생성 (브랜드·테마·프리셋)
4. 사이트별 키 등록 (`PI_API_KEY__<SITE_CD>`) + 재배포
5. **실기기 P0 검증**: Pi Browser 로그인·결제 (핵심가치 — 완료 조건)
6. 법무 문서 브랜드 변수 확인·공지 게시

## 4. 졸업 파이프라인

- **임계 기준(안)**: MAU N명 또는 월 매출 N Pi 도달 (수치는 운영 데이터로 확정)
- **절차**: 템플릿 추출(코드) + `site_cd` 데이터 pg_dump 분리(컷오버 경험 재사용) + 전용 Vercel/Supabase + Pi 앱 그대로 승계(uid 불변 — usr_site_map이 원장)
- 졸업 후 모선과의 계정 연동은 Pi Sign-In(OAuth)으로 유지 가능
