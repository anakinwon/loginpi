# 품질점검기준서 (Quality Inspection Standards)

> **데이터 품질 점검의 기준 문서** — `da-qa-standard-auditor` 에이전트와 `da-qa-checklist` 스킬이 이 문서를 기반으로 점검을 수행한다.
> 규칙의 정본은 [`데이터표준규칙.md`](./데이터표준규칙.md) — 본 문서는 **점검 절차와 체크리스트**를 정의한다.
>
> **최종 갱신**: 2026-06-08

---

## 1. 점검 절차 (4단계)

### Phase 1: 기준 문서 파악
점검 시작 전 반드시 다음 문서를 순서대로 파악한다:
1. `docs/da/데이터표준규칙.md` — 전체 표준 규칙 (정본)
2. 본 문서의 체크리스트 (§3)
3. 직전 점검 보고서 (`docs/da/reports/`) — 반복 위반 패턴 확인

### Phase 2: 점검 수행
점검 대상(데이터 모델, DDL, 메타데이터, 용어 등)에 대해 §3 체크리스트 전 항목을 검토한다.

**점검 영역**:
- **표준사전 준수**: 표준단어·표준도메인·표준용어 등재 여부, 용어 구성 형식
- **명명규칙**: 테이블/컬럼/제약조건/인덱스/트리거 명명 패턴, 소문자 원칙
- **시스템 컬럼**: 4개 컬럼 존재·순서·NOT NULL·DEFAULT
- **논리 삭제**: del_yn + del_dtm, 물리 DELETE 부재
- **타입 규칙**: 날짜타입(UTC)·Y/N 플래그·수치 타입
- **메타데이터 완전성**: 설명, 데이터 타입, 길이, NULL 허용 여부
- **참조 무결성**: FK 정의 여부

### Phase 3: 결과 문서화
- 점검 결과를 §4 보고 양식으로 작성한다
- 정기 점검 보고서는 `docs/da/reports/YYYY-MM-DD_<제목>.md`로 저장한다
- 규칙 자체의 변경이 필요하면 정본(`데이터표준규칙.md`)을 갱신하고 변경 이력에 기록한다

### Phase 4: 후속 조치
- P1 위반: 즉시 수정 (마이그레이션 작성)
- P2 위반: 계획적 수정 (로드맵 등재)
- P3 위반: 개선 과제 (README 로드맵 표)
- 위반 수정 불가/예외 필요 시: **DA 승인 절차** (§2)

## 2. Bypass / 승인 절차

```
DDL 제출 → da-ddl-guard Hook 자동 검사
  ├─ 전 규칙 준수      → ✅ bypass (물리모델 직행)
  └─ 위반 감지         → ⛔ 차단 + DA 통보
       └─ DA 승인 시   → SQL 상단 주석 추가 후 재시도:
                          -- DA-APPROVED: <승인사유> (<YYYY-MM-DD>)
```

- 승인 주석은 SQL 파일에 영구 잔존한다 (감사 추적)
- 승인 권한: DA (프로젝트 오너)

## 3. 점검 체크리스트

### P1 Critical (즉시 수정 — Hook 자동 차단 대상)

**테이블 명명**
- [ ] 테이블명에 도메인 접두사가 있는가? (`sys_`·`brd_`·`std_`·`pi_`·`auth_`·`cod_`·`msg_`·`i18n_`)
- [ ] 테이블명이 소문자 snake_case인가? (대문자·따옴표 식별자 금지)
- [ ] `_sync` 등 비표준 접미사가 없는가?
- [ ] 복수형 테이블명이 없는가?

**시스템 컬럼**
- [ ] 시스템 컬럼 4개(`regr_id`, `reg_dtm`, `modr_id`, `mod_dtm`)가 테이블 **마지막**에 있는가?
- [ ] 시스템 컬럼이 모두 `NOT NULL`인가?
- [ ] `regr_id` / `modr_id`의 타입이 `TEXT`, DEFAULT가 `'ADMIN'`인가?
- [ ] `reg_dtm` / `mod_dtm`의 타입이 `TIMESTAMPTZ`, DEFAULT가 `CURRENT_TIMESTAMP`인가?

**논리 삭제**
- [ ] 운영 테이블에 `del_yn CHAR(1)` 컬럼이 있는가?
- [ ] 신규 테이블에 `del_dtm TIMESTAMPTZ` 컬럼이 있는가?
- [ ] DDL/마이그레이션에 물리 `DELETE FROM` / `DROP TABLE`이 없는가?
- [ ] 삭제 API가 논리 삭제(`del_yn='Y'` + `del_dtm` 기록)를 사용하는가?

**타입 규칙**
- [ ] `_dt`/`_dtm` 컬럼이 날짜 타입(`DATE`/`TIMESTAMPTZ`)인가? (VARCHAR·TEXT·CHAR 금지)
- [ ] 일시 컬럼이 UTC(`TIMESTAMPTZ`)인가? (`TIMESTAMP WITHOUT TIME ZONE` 금지)

### P2 Major (계획적 수정)

**표준사전**
- [ ] 모든 컬럼 물리명이 표준용어 형식(`단어1(_단어n)_도메인`)인가?
- [ ] 컬럼명이 표준도메인 약어로 끝나는가?
- [ ] 사용된 단어가 모두 표준단어 사전(`std_dic`)에 등재되어 있는가?
- [ ] 신규 영문약어가 2~4자인가? (grandfathered 예외 제외)

**명명 일관성**
- [ ] 시스템 컬럼 suffix가 `reg_dtm`/`mod_dtm`인가? (`created_at`/`updated_at` 금지)
- [ ] `mod_dtm` 자동 갱신 트리거가 있는가? (`trg_<테이블>_mod_dtm`)
- [ ] 제약조건 명명이 표준 패턴인가? (`_pkey`·`_key`·`_fkey`·`_check`·`idx_`)

**모델링 절차**
- [ ] Top-down 순서(개념→논리→물리)로 설계되었는가?
- [ ] 주제영역이 정의되어 있는가?
- [ ] 주식별자(PK)가 명시되어 있는가?

### P3 Minor (개선 과제)

- [ ] Y/N 플래그 컬럼이 `CHAR(1)` + CHECK 제약인가?
- [ ] 파일 크기 컬럼이 `BIGINT`인가? (INT는 2GB 한계)
- [ ] SELECT 쿼리에 `.eq('del_yn', 'N')` 필터가 있는가?
- [ ] INSERT 시 `regr_id`에 현재 세션 사용자 ID를 주입하는가?
- [ ] 메타데이터(컬럼 설명, COMMENT)가 작성되어 있는가?
- [ ] FK 참조 무결성이 정의되어 있는가?
