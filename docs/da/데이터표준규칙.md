## 0. 표준 설계 원칙 (Top-down)

DB 설계는 표준설계원칙을 준용하며, 반드시 **Top-down 방식**을 준수한다.

```
개념모델 ──→ 논리모델 ──→ 물리모델
(주제영역)   (표준사전)    (성능 설계)
```

### 0-1. 개념모델 (Conceptual Model)
- **주제영역(Subject Area)을 정의**한다 — 시스템·게시판·표준·Pi결제·인증·메시징 등
- 주제영역별 **주요 엔터티를 도출**한다
- 산출물: 주제영역 정의서, 개념 ERD

### 0-2. 논리모델 (Logical Model)
- **표준사전(§1)을 준수**하여 설계한다
- 설계 항목: 엔터티 · 속성 · 관계 · **논리명 · 물리명 · 주식별자**
- 모든 속성의 물리명은 표준용어 형식(§1-3)을 따라야 한다
- 산출물: 논리 ERD, 엔터티 정의서, 속성 정의서

### 0-3. 물리모델 (Physical Model)
- 테이블 · 컬럼 · PK · Index · 파티션 등 **성능을 고려**하여 설계·구현한다
- DDL은 본 문서의 명명·타입·시스템 컬럼 규칙을 모두 충족해야 한다
- 산출물: 물리 ERD, DDL 스크립트 (`sql/NNN_*.sql`)

### 0-4. Bypass / 승인 절차

```
DDL 작성
  │
  ├─ 규칙 전체 준수 ──→ ✅ bypass — 물리모델 적용 직행
  │
  └─ 규칙 위반 감지 ──→ ⛔ 차단 (da-ddl-guard Hook)
        │
        ├─ DA에게 위반 내역 통보 (자동)
        ├─ DA 검토·승인
        └─ 승인 시 SQL 상단에 주석 추가 후 재시도:
           -- DA-APPROVED: <승인사유> (<YYYY-MM-DD>)
           (주석은 영구 잔존 — 감사 추적)
```

## 1. 표준사전 (Standard Dictionary)

**표준사전 = 표준단어 + 표준도메인 + 표준용어**
관리 위치: `/admin/std/words` · `/admin/std/domains` · `/admin/std/terms` (DB: `std_dic` · `std_dom` · `std_term`)

### 1-1. 표준단어 (Standard Word)
- **한글명 + 축약형 영문약어 쌍**으로 관리한다
- 영문약어는 **2~4자**로 최대한 짧게 한다
- **중복 불허** — 동음이의어(같은 발음·다른 뜻), 이음동의어(다른 표기·같은 뜻) 모두 등록 금지
- 신규 단어 등록 전 기존 사전에서 유사어 검색 필수

> ⚠️ **Grandfathered 예외**: 표준화 이전에 등록된 `pymnt`(5자)·`attch`(5자) 등 기존 약어는 유지한다.
> 2~4자 원칙은 **신규 등록 단어부터** 적용한다.

### 1-2. 표준도메인 (Standard Domain)
- 표준단어를 **먼저 등록**한 후, 그 단어들의 **Type과 Length를 반드시 한정**하여 도메인으로 정의한다
- 도메인 없는 컬럼 타입 임의 지정 금지

| 도메인 약어 | 의미 | 기본 타입 |
|---|---|---|
| `id` | 식별자 | UUID / TEXT |
| `nm` | 이름 | VARCHAR(n) |
| `cd` | 코드 | VARCHAR(n) |
| `yn` | 여부 | CHAR(1) |
| `dtm` | 일시 | TIMESTAMPTZ |
| `dt` | 날짜 | DATE |
| `no` | 번호 | INT |
| `cnt` | 건수 | INT |
| `amt` | 금액 | NUMERIC / BIGINT |
| `sz` | 크기 | BIGINT |
| `ord` | 순서 | INT |
| `url` | URL | TEXT |
| `desc` | 설명 | TEXT |
| `crd` | 좌표값 (WGS84 위·경도) | NUMERIC(11,8) |

### 1-3. 표준용어 (Standard Term)
- 구성 형식: **`표준단어1(_표준단어n)_표준도메인`**
- **반드시 표준도메인으로 끝나야 한다** (예: `reg_dtm`, `theme_cd`, `tip_amt_pi` ❌ → `tip_pi_amt` ✅)
- **미등록 표준단어 사용 금물** — 신규 단어가 필요하면 표준단어 등록 → 승인 → 사용 순서
- 단어와 단어 사이는 **`_`(언더스코어)** 로 구분한다

## 2. 테이블 명명 표준

### 2-1. 도메인 접두사 (필수)

모든 테이블은 주제영역을 나타내는 접두사로 시작해야 한다.

| 접두사 | 주제영역 |
|--------|--------|
| `sys_` | 시스템 공통 (사용자, 권한, 설정) |
| `brd_` | 게시판/커뮤니티 (게시글, 댓글, 첨부) |
| `std_` | 표준 데이터 (단어, 도메인, 용어, 코드) |
| `pi_` | Pi Network 연동 (결제, 트랜잭션) |
| `auth_` | 인증/연동 (연동코드, 토큰) |
| `cod_` | 공통 코드 그룹/상세 |
| `msg_` | 메시징/카페 (PyCafé — Phase 7~9) |
| `i18n_` | 다국어 (locale, 번역) |

**위반 사례 및 수정 이력**

| 변경 전 | 변경 후 | 이유 |
|---------|---------|------|
| `users` | `sys_user` | 시스템 사용자 — `sys_` 접두사 필요 |
| `payments` | `pi_pymnt` | Pi Network 결제 — `pi_` 접두사 필요 |
| `link_codes` | `auth_link_cd` | 인증 연동 코드 — `auth_` 접두사 필요 |
| `std_dic_sync` | `std_dic` | `_sync` 접미사는 비표준 |
| `std_dom_sync` | `std_dom` | 동일 |

### 2-2. 테이블명 형식
- **소문자 스네이크케이스** (`snake_case`) — PostgreSQL 모든 오브젝트는 소문자로 생성·관리 (대문자·따옴표 식별자 금지)
- 복수형 금지 (`posts` → `brd_post`)
- 단어 수 3개 이하 권장
- 약어는 표준단어 사전(§1-1) 등재 약어만 사용

## 3. 컬럼 명명 표준

### 3-1. 시스템 컬럼 (모든 테이블 필수, 반드시 테이블 마지막 4개 컬럼)

```sql
regr_id TEXT        NOT NULL DEFAULT 'ADMIN',              -- 등록자 ID
reg_dtm TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,    -- 등록일시
modr_id TEXT        NOT NULL DEFAULT 'ADMIN',              -- 변경자 ID
mod_dtm TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP     -- 변경일시 (트리거 자동 갱신)
```

**강제 규칙**:
- 4개 컬럼 모두 `NOT NULL` — nullable 금지
- `regr_id` / `modr_id` DEFAULT `'ADMIN'` — 시스템 작업 추적 기본값
- `reg_dtm` / `mod_dtm` DEFAULT `CURRENT_TIMESTAMP` — `NOW()` 동의어이나 표준 표기 준수
- 컬럼 순서: `regr_id → reg_dtm → modr_id → mod_dtm` (이 순서 고정)

**금지 패턴**: `created_at`, `updated_at`, `createdAt`, `updatedAt`, `create_dt`, `update_dt`

### 3-2. PK 컬럼
- UUID 타입 권장: `gen_random_uuid()` DEFAULT
- 컬럼명 패턴: `<엔터티약어>_id` (예: `dic_id`, `dom_id`, `term_id`, `post_id`)

### 3-3. FK 컬럼
- 참조 테이블의 PK 컬럼명을 그대로 사용
- 예: `sys_user.id` 참조 시 FK 컬럼명 = `user_id` 또는 `<역할>_usr_id`

### 3-4. Y/N 플래그 컬럼
```sql
<컬럼명>_yn CHAR(1) NOT NULL DEFAULT 'N' CHECK (<컬럼명>_yn IN ('Y', 'N'))
```
- 반드시 `CHAR(1)` — `TEXT`, `VARCHAR`, `BOOLEAN` 금지
- 주요 플래그: `del_yn` (논리 삭제), `use_yn` (사용 여부), `apv_yn` (승인 여부)

### 3-5. 날짜·일시 컬럼 타입 (강제)
- `_dt` / `_dtm` 컬럼에 **VARCHAR · TEXT · CHAR 사용 불가** — 반드시 날짜 타입 사용
- 일시는 글로벌 용도로 **UTC 필수**: `TIMESTAMPTZ` (TIMESTAMP WITHOUT TIME ZONE 금지)
- 날짜만 필요한 경우: `DATE`

### 3-6. 수치 컬럼 타입 기준

| 범위 | 타입 |
|------|------|
| 카운트/일반 정수 (< 2억) | `INT` |
| 파일 크기, 금액, 큰 수 | `BIGINT` |
| 소수점 없는 금액 | `BIGINT` |
| 소수점 있는 금액 | `NUMERIC(precision, scale)` |

## 4. 논리 삭제 정책

### 4-1. 원칙
- **물리 삭제(`DELETE`) 절대 불가** — 운영 데이터 테이블은 물리 `DELETE` 금지
- 삭제 요건에는 **삭제여부 + 삭제일시** 속성을 반드시 사용:

```sql
del_yn  CHAR(1)     NOT NULL DEFAULT 'N' CHECK (del_yn IN ('Y', 'N')),  -- 삭제여부
del_dtm TIMESTAMPTZ                                                      -- 삭제일시 (삭제 시 기록)
```

- 삭제 시: `UPDATE SET del_yn='Y', del_dtm=CURRENT_TIMESTAMP, mod_dtm=CURRENT_TIMESTAMP, modr_id=<현재사용자>`

> ⚠️ **기존 테이블 경과 조치**: 표준화 이전 테이블은 `del_yn`만 보유.
> `del_dtm`은 **신규 테이블부터 필수** 적용하며, 기존 테이블 소급은 로드맵 과제(`docs/da/README.md`)로 관리한다.

### 4-2. 조회 시 필수 필터
```typescript
.from('std_dic').select('...').eq('del_yn', 'N')
```

### 4-3. 예외
- 순수 로그/이력 테이블 (append-only, `_log`·`_hist` 접미사): 물리 삭제 허용
- 임시 데이터 테이블: 물리 삭제 허용

## 6. 트리거/함수 명명 표준

```sql
-- mod_dtm 자동 갱신 함수
fn_upd_<테이블명>_mod_dtm()

-- mod_dtm 자동 갱신 트리거
trg_<테이블명>_mod_dtm  BEFORE UPDATE ON <테이블명>
```

각 운영 테이블은 UPDATE 시 `mod_dtm`을 자동 갱신하는 트리거를 보유해야 한다.
