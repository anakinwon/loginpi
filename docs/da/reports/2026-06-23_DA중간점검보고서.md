### 상세 분석: sql/026 DELETE 정당성

**컨텍스트**: `stat_revenue_dly` 일별 수익 통계 롤업 함수

```sql
-- fn_stat_revenue_aggregate (일별 통계 재집계, cron 멱등성)
CREATE OR REPLACE FUNCTION fn_stat_revenue_aggregate(p_dt DATE)
AS $$
BEGIN
  -- 1) 기존 당일 통계 제거 (재계산 위해 필수)
  DELETE FROM public.stat_revenue_dly WHERE stat_dt = p_dt;
  
  -- 2) 신규 집계 데이터 삽입
  INSERT INTO stat_revenue_dly (stat_dt, revenue, ...)
  SELECT p_dt, SUM(...), ...
  FROM pi_pymnt
  WHERE DATE(pymnt_dtm) = p_dt;
END $$;
```

**DELETE 정당성**:
1. **통계 테이블 전용 용도** — 사용자 트랜잭션 데이터 아님
2. **cron 멱등성** — 일일 1회 자정 재계산시 이전 집계 제거 필수
3. **감사추적 불필요** — 통계 값 변경이지 실제 거래 변경 아님
4. **물리 DELETE 대안 불가** — `del_yn` 필터 시 재집계 복잡도↑

**평가**: ✅ **예외 인정** — 통계 도메인의 특수성 수용

**단, 향후 권고**:
```sql
-- 더 엄격한 거버넌스를 위한 대안
ALTER TABLE stat_revenue_dly ADD COLUMN del_yn CHAR(1) DEFAULT 'N';

-- DELETE 대신
UPDATE stat_revenue_dly SET del_yn='Y' WHERE stat_dt = p_dt;

-- 조회 시
WHERE del_yn='N' AND stat_dt = ...
```
