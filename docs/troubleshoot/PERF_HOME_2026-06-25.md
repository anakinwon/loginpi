# 성능 문제 기록 — HOME 탭 (2026-06-25)

## 문제 요약
분석 4개 API(`/api/admin/analytics/{orders,usage,performance,pageviews}`)가 공개 전환 후 **응답 캐싱 헤더 미설정**으로 인해 매 Home 방문마다 Supabase 온더플라이 집계 트리거. 게스트 트래픽 폭주 시 DB 부하 증가 및 LCP 악화.

## 증상
- **LCP(Largest Contentful Paint)**: 2.5~4.5초 (목표: < 2.5초)
- **DB 동시성 부하**: cron 배치 + 온디맨드 집계 충돌 시 Supabase 응답 지연
- **익명 트래픽 스크래핑 취약점**: 캐싱 없으면 API 폭주 가능

## 근본 원인
**분석 4개 API 응답 헤더에 Cache-Control 미설정**:
```typescript
// Before: 캐싱 헤더 0
return NextResponse.json({ /* payload */ })

// Result: 매 요청마다 Supabase 풀스캔
// - mps_order: 7일 기간 전 주문행 조회 (RFM 세그먼트 계산)
// - sys_user_actvty_log: 활동 로그 풀스캔 (DAU/WAU/MAU·코호트 계산)
// - stat_pageview: 페이지뷰 풀스캔 (세션 분석)
```

**Home 구조**:
```
Home (TechWhitepaper + UserManual)
├─ StatsDashboard
│  ├─ LazySection: DAU/WAU/MAU 차트 (Plotly)
│  ├─ LazySection: 매출 섹션 (KPI + 차트)
│  └─ LazySection: AnalyticsHub ← 분석 4개 API 호출 (orders/usage/performance/pageviews)
```

AnalyticsHub가 스크롤 진입 시에만 마운트되므로 (LazySection) **초기 로딩 임팩트는 낮으나**, 방문자가 스크롤하면 4개 API가 **캐싱 없이 동시 호출** → DB 스파이크.

## 해결책
**Vercel Fluid 캐싱 적용** — 4개 API 모두 동일 헤더:

### 파일 수정 목록
1. **`src/app/api/admin/analytics/orders/route.ts`** — L217 NextResponse 래핑
2. **`src/app/api/admin/analytics/usage/route.ts`** — L157 NextResponse 래핑
3. **`src/app/api/admin/analytics/performance/route.ts`** — L117 NextResponse 래핑
4. **`src/app/api/admin/analytics/pageviews/route.ts`** — L107 NextResponse 래핑

### 적용 패턴
```typescript
return NextResponse.json(
  { /* payload */ },
  {
    headers: {
      // Vercel edge 캐싱 60분 (모든 visitor 공유)
      'Cache-Control': 's-maxage=3600, max-age=0, stale-while-revalidate=3600',
      // CDN(Cloudflare 등) 호환성
      'CDN-Cache-Control': 'max-age=3600, stale-while-revalidate=3600',
    },
  }
)
```

### 헤더 의미
- **`s-maxage=3600`**: Vercel edge caching 60분 (모든 visitor 공유, Supabase 쿼리 1회만)
- **`max-age=0`**: 브라우저 캐시 미사용 (항상 fresh, 집계 시점 명확)
- **`stale-while-revalidate=3600`**: edge 캐시 만료 후 60분간 stale 값 제공 중 백그라운드 갱신

### 예상 개선
- **Home 초기 로딩**: -60~70% (API 응답 edge에서 즉시 제공)
- **DB 쿼리 빈도**: -97% (1시간당 1회만 실 집계, 나머지는 edge cache)
- **동시 요청 수**: -85% (같은 기간 방문자군은 캐시된 응답 재사용)
- **LCP**: 2.5~4.5초 → 1.8~2.3초 (API 응답 지연 제거)

## 검증 방법
### Pi Browser 실기기 검증
1. Home 탭 진입
2. Chrome DevTools → Network 탭
3. `/api/admin/analytics/orders` 요청 검사
4. **응답 헤더** 확인:
   ```
   cache-control: s-maxage=3600, max-age=0, stale-while-revalidate=3600
   ```
5. 같은 기간 2회 방문 (1시간 이내):
   - 1번째: Network 탭 → 약 800ms~2.5s (실 쿼리)
   - 2번째: 약 0ms (edge cache)
