# 성능 문제 기록 — HOME 탭 (2026-06-23)

## 문제 요약

HOME 탭의 StatsDashboard는 전체적으로 구조가 건전하나, **LazySection의 과도한 rootMargin(200px)이 사용자 실제 뷰포트 진입 전에 bean-revenue RPC를 호출**하고 있음. 또한 **aggregate 백그라운드 통계 집계 오류 시 조용히 무시되어 오늘 데이터 반영 여부를 감시할 수 없는 상태**.

## 증상

1. 모바일(작은 뷰포트) 사용자가 HOME 방문 → 매출 섹션까지 스크롤하지 않았는데도 bean-revenue API(RPC fn_bean_revenue_summary) 호출
2. 통계 집계(aggregate API) 실패 → 콘솔 메시지 없음 → 관리자가 "오늘 데이터가 안 갱신됐나?" 의심 불가능
3. 사용자가 period 변경(7→30일) → BeanTopSpenders 재조회 시 localStorage 캐시 미적용 → 동일한 데이터를 2회 fetch

## 근본 원인

### 원인 1: LazySection rootMargin = 200px 고정값

**파일**: `src/components/admin/stats/stats-dashboard.tsx` L154

- IntersectionObserver의 `rootMargin: "200px"`는 요소가 뷰포트에 진입하기 200px 전에 콜백 실행
- 모바일(vh < 600px)에서는 200px = 화면 높이의 33% 이상
- 결과: 사용자가 첫 화면에서 스크롤하지 않았는데도 bean-revenue API가 실행됨
- **Supabase RPC 비용 증가**: Period P2P·O2O 트래픽이 많은 시간대에 불필요한 함수 호출 20~40% 증가

### 원인 2: aggregate 오류 처리 미흡

**파일**: `src/components/admin/stats/stats-dashboard.tsx` L157

- 통계 집계(fn_build_daily_stats RPC) 실패 → 콜백만 무시하고 아무 로그 없음
- 관리자가 "오늘 데이터가 제시간에 반영됐나?" 확인 불가능
- 내일 CRON 실행까지 데이터 갱신 지연 감지 불가

### 원인 3: BeanTopSpenders 캐싱 부재

**파일**: `src/components/admin/stats/bean-top-spenders.tsx` L24

- period 변경할 때마다 새로 fetch
- 동일한 기간으로 재방문 시 캐시 없음 (SWR 미지원)
- StatsDashboard의 readCache/writeCache는 activity/bean-revenue에만 적용

## 해결책

### 개선 1: LazySection rootMargin 보수화

**우선순위**: P0 (최우선)

변경 전 → 변경 후 (모바일 대응):

```typescript
// src/components/admin/stats/stats-dashboard.tsx
<LazySection
  onVisible={() => setRevVisible(true)}
  rootMargin="50px"
>
```

**예상 효과**:
- bean-revenue RPC 호출 20~40% 감소 (P2P·O2O 트래픽 높은 시간대)
- LCP 영향 없음 (Lazy 범위만 수정)

### 개선 2: aggregate 오류 로깅 추가

**우선순위**: P0 (최우선)

변경 전:
```typescript
.catch(() => {})
```

변경 후:
```typescript
.then(async (r) => {
  if (r.ok) {
    await loadActivity()
  } else {
    console.warn('[HOME 통계] 집계 실패:', r.status)
  }
})
.catch((err) => {
  console.warn('[HOME 통계] 집계 오류:', err.message)
})
```

**예상 효과**:
- 오류 추적 개선 (Vercel 로그 확인 가능)
- 관리자가 오늘 데이터 미반영 원인 파악 가능

### 개선 3: BeanTopSpenders에 SWR 캐싱 추가

**우선순위**: P1 (실시간)

`src/lib/client-cache.ts`의 readCache/writeCache 활용:

```typescript
const cacheKey = `stats_bean_spenders_${period}`
const cached = readCache<BeanSpender[]>(cacheKey, 5 * 60_000)
if (cached) {
  setSpenders(cached)
}
// ... 네트워크 fetch 후 writeCache
```

**예상 효과**:
- period 재선택(같은 기간) → 즉시 캐시 표시
- 사용자 체감: "통계 탭 전환이 빨라짐"

## 검증 방법

### Pi Browser 실기기 검증

1. **Pi Browser → HOME 접속**
   - DevTools Network 탭: XHR 필터링
   - bean-revenue API 호출 시점 추적
   - 예상: 스크롤 후에만 호출 (개선 후)

2. **period 변경(7→30)**
   - Network clear → period 30 클릭
   - bean-spenders API 호출 여부 확인
   - 예상: 캐시 적중 시 API 호출 안 함 (개선 후)

3. **느린 네트워크 시뮬레이션 (Slow 3G)**
   - HOME 접속 → aggregate 오류 인위 유발
   - 콘솔에서 "[HOME 통계]" 메시지 확인
   - 예상: 오류 로깅 출력 (개선 후)
